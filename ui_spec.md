# UI 规格文档

> 存储 UI 相关的 spec 探讨结论、布局约束、组件尺寸标准。
> 基准屏幕宽度：**360dp**（主流中低端手机）。

---

## 一、界面设计层

### 1. 触摸热区与手势

| 规则 | 值 | 依据 |
|---|---|---|
| 最小触控尺寸 | **48dp × 48dp** | 苹果 HIG + Material Design 硬性要求 |
| 长按置顶时间 | 0.5–0.8 秒 | 配合震动/视觉反馈（R4） |
| 滑动删除阈值 | 适度（避免与列表滚动冲突） | R2 点击优先 |
| 键盘弹起 | 输入框自动上移 | `Window.softinput_mode` 配置 |

**当前差距**：
- func_row 图标 28dp、NoteCard 按钮 32dp、标签芯片 28dp → 均不达标
- 需统一提升到 ≥48dp（按钮图标至少 36dp，整体热区通过 padding 补足到 48dp）

### 2. 信息层级与字体

**字体层级**（字号由大到小）：

```
H6 (~20sp)       → 区块标题："未完成""已完成 (3)"
Subtitle1 (~16sp) → 便签卡片标题
Body1 (~14sp)     → 设置页标签、空状态文案
Body2 (~12sp)     → 卡片内容预览、UndoBar 文字
Caption (~12sp)   → 标签芯片、搜索条件标签、func_row 标签
```

**字重与颜色区分**：
- 标题：Subtitle1 + 主题主色（无删除线）
- 正文：Body2 + 默认文字色
- 已完成：Body2 + 删除线 + 透明度降低（非死灰色）
- 标签：Caption + 半透明背景

**当前差距**：
- func_row 标签用 Overline（10sp）→ 应改为 Caption（12sp）

### 3. 卡片内边距与间距

| 位置 | 间距 |
|---|---|
| 卡片内 padding | ≥12dp |
| 标题与内容之间 | ≥8dp |
| 内容与标签组之间 | ≥8dp |
| 标签组与操作按钮之间 | ≥4dp |
| 卡片之间 | 8dp |

### 4. 夜间模式

- **语义颜色**：不仅黑底白字。已完成态文字用更低透明度（如 38%），高优先级标签保持视觉警示（深红背景+浅红文字）
- **实时切换**：切换时无闪烁、无白屏过渡（R11）
- **持久化**：用户选择自动保存（R9），下次打开恢复

**当前状态**：`settings_screen.py` 已实现开关 + 持久化，`app.py` 中 `theme_style = "Light"` 作为默认（R7）。

### 5. 响应式适配

- **宽度**：卡片用 `size_hint_x` 弹性伸缩，不固定像素宽度
- **横竖屏**：保持单列布局，卡片随屏幕变宽
- **FAB 定位**：距右下角固定 16dp（非百分比），保证不同屏幕一致

**当前状态**：卡片已用 `size_hint_x=1`，FAB 待改为固定 dp 定位。

### 6. 空状态处理（R5）

| 场景 | 显示内容 |
|---|---|
| 无便签 | 插画 + "还没有便签，点击右下角 + 创建一个吧" |
| 无标签 | "暂无标签，点击右上角 + 创建" |
| 搜索无结果 | "没有找到匹配的便签" |

**当前状态**：三种空状态文案已实现，插画未实现（低优先级）。

## 三、标签展示方案

**选择：固定宽度截断。**

- 每芯片 `size=(dp(90), dp(32))`，字体 Caption
- 3 个芯片总计 90×3 + 间距 ≈ 278dp，在 360dp 屏幕内不溢出
- 文字超出自动截断，无需 ScrollView 或 "+N" 展开
- 否决方案：水平 ScrollView（嵌套滑动冲突）、"+N" 芯片（过度设计）

## 五、代码实现层

### 7. MVC 分离（R26）

- **UI 层只做展示 + 转发事件**：按钮点击 → 调用 Service → 接收 `(bool, result)` → 更新界面
- **禁止在 UI 代码写 SQL**：数据库操作仅限 `model/database.py` 和 `controller/` 层
- **好处**：测试时可 mock Service，不碰真实数据库（R27）

**当前状态**：`main_screen.py` 通过 `MDApp.get_running_app()` 获取 service，未直接操作数据库。

### 8. 界面状态管理与刷新

**增量更新优先**：
- 数据变更后尽量只更新变化的那张卡片，而非全量 `clear_widgets()` + 重建
- 若必须全量刷新，考虑使用 `RecycleView` 的数据驱动机制减少闪烁

**生命周期保护**（Kivy `self.ids` 陷阱）：
- 在 `on_enter` 中刷新列表，不在 `__init__` 中访问 `self.ids`
- 使用 `if self.ids` 守卫，防止过早访问导致崩溃
- 必要时用 `Clock.schedule_once` 延迟到下一帧

**当前状态**：`main_screen.py` 在 `on_enter` → `refresh_list` 中使用 `clear_widgets()` 全量重建。

### 9. 动画性能（R11）

- **使用 `Animation` 类**：删除「缩小+淡出」0.2–0.3s，新增「底部滑入」
- **动画不阻塞主线程**：数据状态先变更，动画仅负责视觉效果
- **失败回退**：动画执行失败时，数据状态已变更，UI 可直接刷新到最终态

**当前状态**：动画零实现（用户确认非重点，低优先级）。

### 10. UI 自动化测试可寻址性（R25）

- 关键控件起稳定的 `id`：`fab_add`、`input_title`、`input_content`、`chips_box`
- 测试依赖 `id` 定位，不依赖文字（如 `text="添加"`）
- **当前状态**：KV 中关键控件已有 id（如 `title_input`、`content_input`、`note_list`、`fab`）

## 八、排序与置顶行为预期

> 本节定义便签列表排序的**目标行为**，作为后续修复和测试的基准。

### 8.1 置顶排序层级

便签列表从顶到底分为 3 层，**前 2 层不受排序按钮影响**：

```
┌──────────────────────────────┐
│  【手动置顶区】                │  ← pinned_at DESC（最近置顶最上）
│  免疫排序按钮                  │
│  触发：卡片点击置顶按钮         │
├──────────────────────────────┤
│  【标签置顶区】                │  ← 按标签被选为置顶的顺序排列
│  免疫排序按钮                  │     同标签内按 updated_at DESC
│  触发：标签管理页选择标签置顶   │
├──────────────────────────────┤
│  【非置顶区】                  │  ← 受排序按钮影响
│  排序：created_at/updated_at   │     DESC
│  DESC                         │
└──────────────────────────────┘
```

**冲突去重规则**：便签同时满足手动置顶 + 含置顶标签时，**仅出现在手动置顶区**（手动置顶优先级更高）。

**标签置顶区内排序细则**：
- 不同置顶标签之间：按用户在标签管理页**点选置顶标签的顺序**（即 `pinned_tags` JSON 数组的顺序）排列
- 同一置顶标签内的多张便签：按 `updated_at` DESC（最近更新在前）

### 8.2 新便签位置预期

刚刚创建的、未置顶的便签，无论排序偏好设为"按创建时间"还是"按更新时间"，都应出现在**非置顶区的最上方**（即置顶便签之下、其他普通便签之上）。

当前 bug：新便签出现在列表最下方。根因是 Kivy `add_widget(widget, index=0)` 默认行为将每次插入的 widget 放到 children[0]（BoxLayout vertical 的视觉顶部），导致 service 层返回的正序列表在 UI 层被反转。

### 8.3 排序切换行为预期

| 行为 | 预期 |
|------|------|
| Toast 反馈 | 每次切换弹出可见 Toast："按更新时间排序" / "按创建时间排序" |
| 非置顶区 | 按新排序偏好重排 |
| 手动置顶区 | 顺序不变、无闪烁 |
| 标签置顶区 | 顺序不变、无闪烁 |
| 标题栏/功能栏 | 不触发可见刷新（排序切换不应引起标题栏区域重新渲染） |

### 8.4 性能边界（已确认）

删除、完成、置顶、撤销操作使用 `refresh_list()` 全量重建卡片是**当前最优方案**。增量更新（`remove_widget` + `add_widget` 在容器间移动卡片）已于 2026-06-24 尝试并回退（见 `correct_book.md` §7.3.4），原因是 SQL 瓶颈消除后，widget 树移动操作的开销超过了全量重建。

以下操作继续使用 `refresh_list()`：
- 创建便签（含 `_add_new_card` 增量添加，该路径已优化）
- 编辑便签（就地更新 card 属性，不触发 `refresh_list`）
- 删除/完成/置顶/撤销 → `refresh_list()`

以下操作应使用 `_reorder_cards()` 就地重排：
- 排序切换（无搜索参数时）
- 搜索取消/清除 → 应复用卡片而非全量重建（待优化）
