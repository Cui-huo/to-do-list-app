## 二、开发与架构层规则

> 本层规则面向**代码设计与架构**，旨在保证软件的可维护性、可扩展性、可测试性。

### R12. 伪代码决策树 — 复杂逻辑必须可验证
当排序、搜索或过滤涉及 3 个以上条件组合时，禁止纯自然语言。
必须提供 SQL 片段、Python 伪代码或流程图，明确优先级、边界值和兜底策略。

示例 — 未完成便签排序：
```python
ORDER BY
  CASE WHEN is_pinned=1 THEN 0 ELSE 1 END ASC,   -- 手动置顶最前
  CASE WHEN tag IN pinned_tags THEN 0 ELSE 1 END, -- 标签置顶次之
  CASE WHEN position > 0 THEN -position END DESC, -- 拖拽排序
  {sort_preference} DESC                           -- 时间偏好
```

### R13. 临时状态生命周期 — 内存数据必须声明存活边界
凡是不持久化到数据库的临时数据（撤销缓存、拖拽中间态、搜索关键词），必须明确：
- 存储位置
- 存活时长
- 应用重启/崩溃后的行为

禁止模糊表述为"暂存"或"临时保留"。
- ✅ "撤销数据仅存在于 Python 进程内存。应用被杀或强制关闭后永久丢失，UndoBar 不再显示。首次删除时 Toast 提示「撤销仅在应用运行期间有效」"

### R14. 校验分层 — 前端实时 + 后端提交
每个输入字段的校验必须拆分为两层：

| 层 | 时机 | 用途 |
|---|---|---|
| 前端实时 | 输入过程中 | 字符计数、格式高亮、空白拦截 |
| 后端提交 | 点击保存时 | 唯一性检查、关联完整性、重名检测 |

禁止将两者混为一谈——"标签名不可重名"不能等到点保存才报错。
- ✅ 前端实时："已输入 X/10 字符"；输入纯空白时边框变红
- ✅ 后端提交：重名时弹窗 "标签「XXX」已存在"，自动聚焦输入框

**补充 — 前端复现后端校验**：
后端提交的校验逻辑必须在前端同步复现（后端是最终防线，前端是用户体验）。例如后端有 `title ≤ 50` 约束，前端输入框也应限制 50 字符，并在超出时实时提示。

### R15. 每功能四段式 — 每个功能必须写清
```
输入 → 校验 → 输出 → 副作用
```
缺一不可。

### R16. "明确不做"清单
用独立章节列出坚决不做的事项，防止需求蔓延。

### R17. 数据模型字段完整
每个字段必须标注：类型、约束、默认值、NULL 含义、关联关系。

### R20. 环境隔离 — 项目依赖仅限项目目录
所有开发环境（Python 版本、pip 包、虚拟环境）必须安装在项目目录内，禁止修改系统全局环境。
- ❌ 使用 `winget install Python.Python.3.12` 安装到系统目录
- ❌ 使用 `pip install` 修改全局 site-packages
- ✅ 将 Python embeddable 包解压到 `.python312/` 项目子目录
- ✅ 所有 pip 包安装在项目内的嵌入式 Python 中

**补充 — 导入时禁止耗时操作**：
禁止在 `__init__.py` 中执行耗时操作（如数据库初始化、加载外部资源），避免模块导入时阻塞。

### R24. 状态流转一致性
任何状态变更操作（如标记完成/取消完成、置顶/取消置顶）必须定义：
1. **前置条件**（什么状态下允许执行）
2. **后置条件**（执行后状态如何变化）
3. **关联副作用**（排序是否刷新、UI 动画后是否重新渲染）

若前置条件不满足，返回明确失败原因，不允许静默跳过或部分执行。

示例 — 便签完成流程：
```
前置条件：is_completed == 0
后置条件：is_completed = 1, completed_at = now, position = 0
副作用：完成后自动从"未完成列表"消失，出现在"已完成列表"顶部，执行淡出→淡入动画
失败原因：若 is_completed 已为 1，返回 "该便签已完成，无需重复操作"
```

### R26. 统一返回结构
所有 Service 层公开方法必须返回元组 `(success: bool, result: Any)`，其中 `success` 为 `True` 时 `result` 为有效数据，为 `False` 时 `result` 为错误信息字符串。禁止返回 `None` 表示失败或 `raise` 未经捕获的异常。

示例：
```python
def add_tag(name: str) -> tuple[bool, int | str]:
    if not name or len(name) > 10:
        return False, "标签名 1-10 个字符"
    if self.tag_repo.exists(name):
        return False, f"标签「{name}」已存在"
    tag_id = self.tag_repo.create(name)
    return True, tag_id
```

## 三、测试与质量保障层规则

> 本层规则面向**测试策略与质量保障**，旨在保证代码正确性、可回归性和测试可维护性。

### R18. 测试先行
在规格中给出测试目录结构、隔离策略（`:memory:`）、必须覆盖的 P0 模块。

### R19. DoD 可勾选
每个开发阶段定义清晰的 "Definition of Done" 清单，用于验收。

### R25. 测试用例可追溯
每个测试函数（或测试类）必须在命名或注释中明确标注其对应的 Spec 条目编号（如 `spec.md §5.1` 或 `rules.md R8`）。AI 生成测试代码时自动添加此标注。

示例：
```python
def test_tag_name_length_limit():
    """R8: 标签名 1-10 字符，超限时拒绝"""
    # 测试逻辑...
```

### R27. 测试数据库隔离
单元测试统一使用 SQLite `:memory:` 数据库，确保每个测试互相隔离、运行速度快。集成测试使用临时文件数据库（`tempfile.NamedTemporaryFile`），测试结束后自动清理。禁止单元测试连接生产数据库或持久化文件。

### R29. 时间戳竞争修复 — 连续操作间插入 `time.sleep`
当测试中连续多次调用同一 Service 方法（如 `create()` → `update()` 或 `create()` → `create()`），且断言依赖两操作的 `updated_at` / `created_at` / `completed_at` 时间戳不同时，必须使用 `time.sleep(0.001)` 插入 1ms 延迟。

**根因**：`datetime.now().isoformat()` 精度到微秒，但 `:memory:` SQLite 上连续操作极快，可能落在同一微秒内，导致时间戳完全相同 → 排序断言不稳定、`!=` 断言失败。

**适用范围**：
- 断言 `note.updated_at != updated.updated_at` 的更新测试
- 断言 `ORDER BY updated_at DESC` 排序结果的测试
- 断言 `created_at` / `completed_at` 时序差异的任何测试

**备选方案**：当排序字段等值时用 `id` 做二级排序（如 `ORDER BY updated_at DESC, id DESC`），但 `time.sleep` 更简单直接。

```python
# ✅ 正确
n1 = svc.create(title="第一", content="内容1")
time.sleep(0.001)
n2 = svc.create(title="第二", content="内容2")
notes = svc.get_incomplete()
assert notes[0].title == "第二"

# ❌ 错误 — 可能同微秒，排序不稳定
n1 = svc.create(title="第一", content="内容1")
n2 = svc.create(title="第二", content="内容2")
notes = svc.get_incomplete()
assert notes[0].title == "第二"
```

## 四、AI 行为约束层规则

> 本层规则面向 **AI 辅助编码行为**，旨在防止 AI 过度发挥、擅自补全，确保生成代码严格符合 Spec。

### R22. AI 生成代码边界
AI 仅允许生成 `spec.md` 和 `rules.md` 中明确定义的功能代码。不得私自添加未在 Spec 中声明的"辅助功能""扩展特性""隐藏彩蛋"等。若发现 Spec 缺失而导致无法实现，必须主动提问澄清，不得自行"补全"。

**扩展 — 遇到以下任一情况，必须先问后做，严禁自行决定**：
1. 边界条件不清晰（如"超出上限时是拒绝还是静默截断"）
2. 处理逻辑未定义（如"同时存在冲突条件时谁优先"）
3. 功能需求模糊（如"删除后是软删还是硬删"）
4. 多个合理方案可选（如"用弹窗还是 Toast"）
- ✅ AI 发现"标签重名时未定义行为" → 主动提问"重名时拒绝创建还是允许？弹窗提示？"，等待答复后再编码
- ❌ AI 擅自选择"静默跳过不存在的标签"而未确认是跳过还是报错
- ❌ AI 自行添加"便签历史版本记录"功能（Spec 未定义）

### R30. Bug 修复全局排查 — 同类错误一次修完
修复 bug 时，禁止只修复报错点。必须用 grep 搜索整个代码库，找出所有同类错误模式，一次性全部修复。

- ❌ `_update_theme_colors()` 报 `NameError: MDApp` → 只在这一个方法加 import
- ✅ `_update_theme_colors()` 报错 → grep 全项目 `MDApp.get_running_app()`，检查每处是否有对应 import，发现另一文件同样缺 import，一起修复

**How to apply:** 定位 bug 根因后，提取错误模式（如"方法内调用 X 但缺少 import"），grep 全局所有相同调用点，逐处验证，批量修复。

### R32. 根因优先 — 先理解再修改
解决问题时，必须先定位根本原因，再动手修改。禁止凭猜测、直觉或"试试看"的方式修改代码。

- ❌ 撤销后标题变随机 → 猜测"可能是某处忘了赋值" → 直接加赋值语句
- ✅ 撤销后标题变随机 → 追踪撤销完整调用链 → 定位 undo 缓存存储/恢复时 title 字段丢失 → 精确修复

**Why:** 凭猜测修改可能掩盖真正 bug、引入新问题、或只修复表象而非根因。

**How to apply:** 遇到 bug 时，先追踪完整数据流（输入 → 处理 → 输出），用日志或断点定位异常发生的确切位置，确认根因后再修改。

### R33. Spec-Code 双向同步 — 改一端必须确认另一端
spec.md 与代码必须保持一致性。每次修改一端后，必须主动询问用户是否同步修改另一端。

- spec.md 修改后 → 主动询问「spec 已更新，需要同步修改对应的代码吗？」
- 代码层面功能调整后 → 主动询问「代码已修改，需要同步更新 spec.md 吗？」

**Why:** spec 与代码不一致会导致后续开发决策错误、测试用例偏离预期、新人理解混乱。本次就出现了「spec 写对了（title=原标题）但代码没跟上（_random_title）」的情况。

**How to apply:** 每次完成 spec 或代码的修改后，在最终回复中明确向用户提出同步确认问题。不允许自行决定「不需要改」而跳过。

### R34. 错题本 — 问题解决后记录归档
每次解决完一个技术问题后，必须主动询问：
1. 「问题是否已解决？」
2. 若已解决 →「是否需要保存到错题本 `correct_book.md`？」

错题本记录内容按**对话迭代过程**组织，保留用户与 AI 之间的完整交互链路：

1. **问题描述与背景** — 清晰详细的问题描述 + 发生背景
2. **解决过程（迭代循环）** — 按以下格式记录每一步：

```
   【用户要求】← 用户提出的原始需求和修改指令
       ↓
   【尝试方案】← AI 定位问题根因，提出修复方案并执行
       ↓
   【用户反馈】← 用户报告修复结果（成功/失败/新的要求）
       ↓  （若失败，重复上述循环直到成功）
   【最终方案】← 经过验证的正确解决方案
```

每次尝试必须标注：失败原因、关键发现、以及如何导向下一步。

3. **日期** — 问题解决的日期
4. **模型名称** — 当前与用户对话的底层大模型准确 ID（如 `DeepSeek V4 Pro`）。注意：这里是底层大模型名称，不是工具/平台名称（如 `Qwen Code`）

**Why:** 错题本是个人技术成长的重要资产——记录错误比记录成功更有价值。定期回顾避免重复踩坑。

**How to apply:** 每次 bug 修复、技术难题解决后，在回复末尾附加以上两个追问。待用户确认后，将内容追加写入项目根目录的 `correct_book.md`。

### R35. 内置优先 — 优先使用框架/库内置功能
使用任何技术栈、框架、库、工具进行开发时，必须优先寻找和使用其内置的组件、函数、工具完成代码实现。只有在内置功能确实无法满足需求（如功能不存在、性能瓶颈、完全不适合场景）且经过明确评估后，才允许自定义实现。

- ❌ Kivy 有 `ScrollView` 却手写触摸滚动逻辑
- ❌ Python `datetime` 可以处理时区却引入 `pytz` 库
- ❌ SQLite 支持 UPSERT 却手写 INSERT OR UPDATE 判断逻辑
- ✅ 优先查阅官方文档寻找内置 API → 确认无内置方案 → 才自定义

**Why:** 内置功能经过充分测试、与框架深度集成、维护成本低。自定义造成重复造轮子、bug 率上升、后续升级困难。

**How to apply:** 编码前先查阅官方文档/源码，确认是否有内置方案。需要自定义时在代码注释中标注原因。

### R36. 问题不清晰先确认 — 禁止猜测式修复
进行 bug 修复或其他任何编程任务时，若对问题的理解存在任何不清晰之处（包括但不限于：问题表现、期望行为、涉及范围），禁止自行猜测后直接动手修改。必须先向用户逐条确认不清晰的点，彻底弄清楚问题后再进行修复。

- ❌ 用户说"按钮离得有点远" → AI 猜测是垂直间距问题 → 直接改 func_row 高度
- ✅ 用户说"按钮离得有点远" → AI 反问"是指图标和文字之间的垂直间距，还是四个功能按钮之间的水平间距？" → 确认后再修改
- ❌ 用户反馈"标题偏下" → AI 猜测 pos_hint 不生效 → 直接包 FloatLayout
- ✅ 用户反馈"标题偏下" → AI 先用截图/坐标确认偏了多少，确认根因 → 再修改

**Why:** 猜测式修复有三大危害：① 可能修错地方，引入新问题；② 浪费用户时间来回纠正；③ 掩盖真正的根因，让问题变得更难追踪。

**How to apply:** 接到任务后，先用自己的话复述对问题的理解，列出所有不确定的点，逐条向用户确认。用户确认完毕后，再动手修改。

### R37. 已有实现直接复用 — 禁止重复造轮子
在同一项目中，已经正确实现过的类似功能、函数、组件，**禁止重写**。必须直接参照已有正确实现使用。

- ❌ 3 个页面各自手写不同版本的 `_toast()`，一个用 `MDSnackbar().text = xxx`（不生效），另一个用 `MDSnackbar(MDLabel(...))`（正确）
- ❌ 写新页面时自己造一套弹窗，不看已有的 `build_confirm_dialog`
- ✅ 发现 `main_screen.py` 的 `_toast` 已验证可用 → 其他页面直接复制同一份实现
- ✅ 完成一项之前做过的类似任务时，**先查阅错题本 `correct_book.md`**，避免踩同样的坑

**Why:** 同一项目中存在多个正确实现时，新代码容易参照错误版本；同一功能反复重写浪费时间和引入不一致的 bug。本规则与 R30（Bug修复全局排查）互补——R30 面向已知 bug 的全局修复，R37 面向新功能实现时优先查找已有正确范本。

**How to apply:** 实现任何功能前，先 grep 全项目查找同名/相似函数，确认是否存在已验证的正确实现。如有，直接复用。开始前必须先读 `correct_book.md` 中与本任务相关的条目。

### R38. 每次修改必做性能检查 — 确保操作流畅度
AI 对代码的每一次修改（新增功能、重构、优化、bug 修复），必须在完成功能验证后执行性能检查，确保应用操作的流畅度不受影响。

**检查项**：
| 操作场景 | 检查点 | 阈值 |
|---|---|---|
| 创建便签 | 从点击保存到卡片出现在列表 | 无明显卡顿（主观感知 < 300ms） |
| 编辑便签 | 从点击保存到卡片属性更新 | 无闪烁、无全量重建 |
| 删除便签 | 从确认删除到卡片消失 | 无卡顿 |
| 标记完成/取消完成 | 卡片状态切换 | 无卡顿 |
| 置顶/取消置顶 | 卡片跨区域移动 | 无卡顿、无闪烁 |
| 排序切换 | 非置顶区重排 | 置顶区无闪烁、标题栏不刷新 |
| 搜索筛选 | 列表交叉淡入淡出 | 0.2s 内完成 |
| 标签管理 | 批量删除、置顶切换 | 无 O(N²) 重建 |
| 撤销操作 | 卡片恢复到列表 | 无卡顿 |

**检查方法**：
1. 修改代码后运行全量测试 `pytest app_tool/tests/ -v`，确认无回归
2. 人工或自动化检查关键操作路径中是否存在以下反模式：
   - 单次操作触发 `clear_widgets()` + 全量重建所有卡片
   - 循环内逐条 SQL 查询（N+1）
   - 同一帧内 `Clock.schedule_once` 堆积超过 10 个回调
   - O(N²) widget 树遍历

**Why:** 2026-06-24 审查发现项目存在 3 层性能瓶颈（全量销毁重建、N+1 SQL、Clock 回调堆积）。功能正确性通过后，性能是用户体验的最后一道防线。每次修改都应确保不引入新的性能退化。

**How to apply:** 每次完成代码修改后，在回复中报告：① 全量测试结果（通过数/失败数）② 是否引入 `clear_widgets()` 全量重建 ③ 是否增加新的 SQL 查询或 Clock 回调。若引入性能退化，必须在同一轮修改中解决。

### R39. UI 初始化禁止裸数据库访问 — 必须防御性兜底

UI 组件的 `__init__`、`on_enter`、`on_kv_post` 等生命周期方法中，**禁止直接访问数据库而不做异常防护**。所有数据库读操作必须包裹在 `try/except` 中，确保在测试环境（`:memory:` 连接已关闭或 `None`）中不会崩溃。

- ❌ `_load_username()` 中 `app.db_conn.execute(...)` 无 try/except → 测试环境 `db_conn` 为已关闭连接时崩溃
- ✅ `try: row = app.db_conn.execute(...) except Exception: pass`
- ❌ `on_enter` 中无条件调用 `refresh_list()` 触发 `get_incomplete()` → 无数据库时崩溃
- ✅ `on_enter` 中先检查 `app.db_conn` 可用再刷新列表

**适用场景（R39-A 增强安全）：** 除业务代码防御外，**测试** `conftest.py` 中 function 级别的 `kivy_app_instance` fixture **必须**在每次返回前执行 `kivy_app.db_conn = None` 重置，作为全局兜底——防止前一测试函数体内 `app.db_conn = db_conn` 赋值在异常路径未恢复时污染后续测试。

```python
# ✅ conftest.py — 强制重置
@pytest.fixture
def kivy_app_instance(kivy_app):
    from kivy.app import App
    if App.get_running_app() is None:
        App._running_app = kivy_app
    kivy_app.db_conn = None  # ← 强制重置，防污染
    return kivy_app
```

**Why:** 2026-06-25 `test_ui.py` 56 个测试中 27 个因 `_load_username()` 裸调用 `app.db_conn.execute()` 崩溃（21 ERROR + 6 FAILED）。根因是 `kivy_app.db_conn` 被前序测试污染为已关闭连接，而 `if app and app.db_conn:` 守卫对已关闭连接对象返回 `True`（已关闭连接 ≠ `None`）。修复方案：① 生产代码加 `try/except`（R39 强制）② conftest 每次重置（R39-A 增强）。修复后 56/56 全绿。

**How to apply:** 编写任何 UI 组件生命周期方法时，先确认是否有数据库访问。若有，必须 `try/except`。编写测试 conftest 时，`kivy_app_instance` 必须重置 `db_conn = None`。

### R41. 先查后做 — 解决问题前必须查阅现有知识

接到任何不熟悉的技术任务时，必须遵循"先查 → 再定方案 → 最后动手"三步流程，禁止凭猜测从零试错。

**三步流程**：

| 步骤 | 操作 | 检查项 |
|---|---|---|
| **先查** | 查阅现有知识库 | ① `skill` 列表中有无匹配 skill → invoke 它<br>② 项目 memory 中有无相关记录<br>③ `correct_book.md` 中是否有同类问题记录<br>④ 公开最佳实践（官方文档、GitHub 模板、成熟方案） |
| **再定方案** | 制定修复方案 | 基于查阅结果，列出每步操作和预期结果，提交用户确认 |
| **最后动手** | 执行 + 验证 | 逐步执行方案，每步验证结果 |

- ❌ 构建 APK 失败 → 不知道 buildozer 与 Python 3.12 的兼容性问题 → 凭猜测先试 `setuptools` → 失败 → 再降级 Python → 浪费 2 轮构建
- ✅ 构建 APK 失败 → 先 invoke `buildozer-github-actions` skill（项目中已有）→ 读到"必须用 Python 3.11"→ 直接修改 → 1 轮通过
- ❌ 遇到不熟悉的框架 API → 自己设计参数组合 → 多次试错
- ✅ 遇到不熟悉的框架 API → 先查官方文档 → 参考已有项目中的正确用法 → 一次写对

**Why:** 2026-06-26 APK 构建连续 4 轮失败——项目已有 `buildozer-github-actions` skill 明确写了 Python 3.11 要求和 setuptools shim 无效的结论，AI 却没有 invoke 它，凭猜测从零试错。用户批评："已有多年成熟方法，为什么不查看现有最佳实践？"

**How to apply:**
1. 看到不熟悉的任务 → 第一反应不是"怎么修"，而是"有没有现成知识"
2. 扫描 `<available_skills>` 列表，匹配相关 skill 并 invoke
3. 检查项目 memory 和 `correct_book.md`
4. 以上都查完后，再制定方案并请用户确认
5. 此规则适用于所有领域：CI/CD、框架 API、库函数、系统配置等

### R42. 知识来源分级 — 按置信度决策，最高置信度交叉验证

解决问题时必须按知识来源的可靠性分级决策，禁止将低置信度来源当作"最佳实践"引用。

**知识来源置信度分级**：

| 置信度 | 来源编号 | 来源 | 说明 |
|--------|---------|------|------|
| **高** | ① | 官方技术文档 | python.org、kivy.org、buildozer.readthedocs.io 等，权威正确 |
| **高** | ② | AI 自有知识库 | 预训练模型中的通用编程知识，覆盖面广 |
| **高** | ③ | 社区/技术网站 | Stack Overflow、GitHub Issues、CSDN 等，实战验证 |
| **高** | ⑥ | 错题本 correct_book.md | 已解决问题的完整迭代记录，实战验证过 |
| **高** | ⑩ | 用户当前对话明确指示 | 当前任务最高权威，必须优先执行 |
| **高** | ⑨ | Git 提交历史 | 事实记录，代码变更的唯一溯源 |
| **中** | ④ | 项目 spec/rules 文档 | 人工编写，可能与代码不同步 |
| **中** | ⑤ | 项目手工 skill（非 auto-） | 经验证后沉淀，但场景可能变化 |
| **低** | ⑦ | Qwen Code 自动生成 skill（auto-skill-） | 从对话自动提炼，未经人工审核，可能含试错中的错误结论 |

**决策规则**：

| 场景 | 规则 |
|------|------|
| **首次尝试** | 优先用高置信度知识（①②③⑥⑨⑩）+ 已有项目知识解决问题 |
| **首次失败** | 必须升级到**最高置信度验证**：① + ② + ③ 三方交叉验证，全部通过后方可采纳结论 |

| **引用来源** | 给出结论时必须标注来源于哪个置信度层级（如"③ Stack Overflow 方案"） |

- ❌ 构建 APK 失败 → 直接引用 auto-skill ⑦ → 没有检查官方文档 ① → 没有搜索社区方案 ③ → 反复试错

- ❌ 计划模式中直接提交方案 → 未交叉验证 → 用户质疑"你参考的真的是行业最佳实践吗"
- ✅ 计划模式 → 方案包含"已验证：① python.org 指出 X、③ 5 篇社区文章一致推荐 Y、② 原理上 Z 是合理的"

**Why:** 2026-06-26 APK 构建中 AI 引用的"最佳实践"实际来自 auto-skill ⑦（对话历史回放），用户质问"你参考的真的是行业最佳实践吗"后搜索 ③ 发现真正的社区共识（libffi-dev、cmake 等），暴露了低置信度来源被当作权威引用的根本问题。

**How to apply:**
1. 每次提出方案前，标注引用的知识来源编号（如"①②③交叉验证"）
2. 方案失败一次后，必须启动 ①②③ 交叉验证
3. 计划模式输出方案时，必须附带交叉验证证据（官方文档链接 + 社区讨论截图/引用 + 原理分析）

### R40. 排除法定位 — 超过 2 次定位失败时结构化追问特征

当用户问题描述足够清晰，但 AI 超过 2 次尝试仍无法准确定位根因时，**必须暂停直接推理**，转而使用排除法：通过结构化追问用户具体的行为特征细节，逐模块排除无关代码路径，缩小定位范围。

- ❌ 连续 5+ 轮都在同一个模块（如 `add_widget`）中打转 → 反复验证同一个假设，陷入死循环
- ❌ 读了半天代码，发现推理建立在未经验证的假设上（如"children[0]=TOP"）→ 假设本身就是错的
- ✅ 2 次尝试失败后 → 停止代码推理 → 列出当前所有假设 → 逐条向用户确认 → 通过具体特征排除假设
- ✅ 列出可能涉及的代码模块（A / B / C），设计具体问题让用户通过操作区分是哪个模块的问题

**排除法追问模板**：
| 特征维度 | 排除规则 | 具体问题 |
|---|---|---|
| 操作路径 | 创建 vs 置顶 vs 排序 走不同代码 | 是创建新便签立刻反？还是置顶/排序后才反？ |
| 时期 | 首次启动 vs 操作后 判断是否累积 | 重启后无任何操作，仅创建便签，顺序对吗？ |
| 范围 | 置顶区 vs 非置顶区 区分 SQL tier 还是 UI | 只置顶区反了，还是全部反了？ |
| 数据源 | DB 查询 vs UI 渲染 | Service 层返回顺序查过吗？ |

**Why:** 2026-06-25 便签卡片顺序反转 bug，AI 连续多轮在 `add_widget` 和代码逻辑层打转，全部基于"children[0]=视觉顶部"的错误假设。用户已提供充分信息（"整列完全反转"），但 AI 未能在前 2 轮后停止推理、追问具体特征来排除无关方向，直至阅读 Kivy 源码 `_iterate_layout`（L285 `y = padding_bottom`）才发现真正根因是 children[0]=视觉底部。

**How to apply:**
1. 每次定位失败后计数，超过 2 次 → 停止代码推理
2. 列出当前依赖的所有假设，标记为"已验证"或"未验证"
3. 对未验证假设，设计具体问题向用户确认
4. 通过用户回答排除无关模块，缩小范围后再深入源码验证
