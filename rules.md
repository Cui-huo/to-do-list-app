# 设计规则 (Rules)

> 本文档定义便签 APP 的全局设计约束。所有开发、评审、AI 辅助编码均须遵守。
> 与 `spec.md`（功能规格）、`skills.md`（可复用流程）互补。
> 规则自动合并去重，以编号优先。

---

## 一、交互层规则

### R1. ID 不可见
用户界面只展示可读名称，底层 ID 对用户完全不可见。
- ❌ 错误提示 "标签 ID #42 不存在"
- ✅ 错误提示 "标签「工作」不存在"

### R2. 点击优先 — 能点就不打，能滑就不点
除非必须输入文字或编辑内容，永远优先选择点击、滑动、选择器等前端交互。
- ❌ 让用户手动输入日期 "2024-06-14"
- ✅ 日期选择器，点两下搞定

### R3. 安全确认 — 不可逆操作必须二次确认
所有删除、清空、覆盖、退出等不可逆操作，必须弹出确认对话框并明确告知后果。
- ❌ 点击删除按钮，数据瞬间消失
- ✅ 「确认删除便签「买牛奶」？可在 12 小时内撤销。」→ [取消] [确认删除]

### R4. 即时反馈 — 每个操作必须有 UI 反馈
用户做了任何事，系统必须在 100ms 内给出可见反馈，不能让用户猜测"刚才到底成功了没"。
- ❌ 点击保存后页面毫无变化
- ✅ 保存按钮变灰 → 显示"已保存" SnackBar → 列表自动刷新

### R5. 空态处理 — 优雅处理空白和异常
所有列表、搜索结果、筛选条件都必须定义"空的时候显示什么"，不能留白或崩溃。
- ❌ 没有便签时显示一片空白
- ✅ 空状态插画 + "还没有便签，点击右下角 + 创建一个吧"

### R6. 幂等拒绝 — 重复操作给出明确拒绝
同一对象不能重复执行互斥操作，系统要明确返回失败并告知原因。
- ❌ 对已置顶便签再次置顶，系统无任何反应
- ✅ 弹出提示 "该便签已置顶"

### R7. 默认优先 — 替用户做最合理的选择
每个配置项必须有默认值，用户不设置也能正常使用。
- ❌ 首次打开要求先选择排序偏好才能进入
- ✅ 默认按更新时间排序，设置项自动持久化

### R8. 边界显式 — 所有数字约束必须显式定义
长度、数量、时间、容量等限制必须写在规格里，并配套超限处理策略。
- ❌ "标签名不要太长"
- ✅ "标签名 1–10 字符，超出时输入框禁止继续输入并提示「最多 10 个字符」"

### R9. 持久化 — 用户的选择必须记住
用户的偏好设置、排序方式、主题模式等必须本地持久化，下次打开自动恢复。
- ❌ 每次打开应用都重置为浅色模式
- ✅ 关闭前是夜间模式，下次打开仍是夜间模式

### R10. 可撤销 — 误操作必须可恢复
删除、修改等操作尽量提供撤销机制，而不是一删了之。
- ❌ 删除后数据永久消失
- ✅ 底部弹出 UndoBar「已删除 [撤销]」，保留 12 小时

## 二、规格文档规则

### R11. 视觉过渡 — 状态变更必须定义动画
任何改变数据状态的操作，规格中必须同时描述动画/过渡效果和最终数据态。
禁止只写 `设置 is_completed=1` 而忽略 UI 如何平滑变化。
- ✅ "点击完成后，卡片执行 0.3s 透明度渐变 + 位移动画，动画结束后才刷新列表排序"

### R12. 伪代码决策树 — 复杂逻辑必须可验证
当排序、搜索或过滤涉及 3 个以上条件组合时，禁止纯自然语言。
必须提供 SQL 片段、Python 伪代码或流程图，明确优先级、边界值和兜底策略。

示例 — 未完成便签排序：
```python
ORDER BY
  CASE WHEN is_pinned=1 THEN 0 ELSE 1 END ASC,   -- 手动置顶最前
  CASE WHEN tag IN pinned_tags THEN 0 ELSE 1 END, -- 标签置顶次之
  CASE WHEN position > 0 THEN -position END DESC, -- 拖拽排序
  {sort_preference} DESC                           -- 时间偏好
```

### R13. 临时状态生命周期 — 内存数据必须声明存活边界
凡是不持久化到数据库的临时数据（撤销缓存、拖拽中间态、搜索关键词），必须明确：
- 存储位置
- 存活时长
- 应用重启/崩溃后的行为

禁止模糊表述为"暂存"或"临时保留"。
- ✅ "撤销数据仅存在于 Python 进程内存。应用被杀或强制关闭后永久丢失，UndoBar 不再显示。首次删除时 Toast 提示「撤销仅在应用运行期间有效」"

### R14. 校验分层 — 前端实时 + 后端提交
每个输入字段的校验必须拆分为两层：

| 层 | 时机 | 用途 |
|---|---|---|
| 前端实时 | 输入过程中 | 字符计数、格式高亮、空白拦截 |
| 后端提交 | 点击保存时 | 唯一性检查、关联完整性、重名检测 |

禁止将两者混为一谈——"标签名不可重名"不能等到点保存才报错。
- ✅ 前端实时："已输入 X/10 字符"；输入纯空白时边框变红
- ✅ 后端提交：重名时弹窗 "标签「XXX」已存在"，自动聚焦输入框

### R15. 每功能四段式 — 每个功能必须写清
```
输入 → 校验 → 输出 → 副作用
```
缺一不可。

### R16. "明确不做"清单
用独立章节列出坚决不做的事项，防止需求蔓延。

### R17. 数据模型字段完整
每个字段必须标注：类型、约束、默认值、NULL 含义、关联关系。

### R18. 测试先行
在规格中给出测试目录结构、隔离策略（`:memory:`）、必须覆盖的 P0 模块。

### R19. DoD 可勾选
每个开发阶段定义清晰的 "Definition of Done" 清单，用于验收。

### R20. 环境隔离 — 项目依赖仅限项目目录
所有开发环境（Python 版本、pip 包、虚拟环境）必须安装在项目目录内，禁止修改系统全局环境。
- ❌ 使用 `winget install Python.Python.3.12` 安装到系统目录
- ❌ 使用 `pip install` 修改全局 site-packages
- ✅ 将 Python embeddable 包解压到 `.python312/` 项目子目录
- ✅ 所有 pip 包安装在项目内的嵌入式 Python 中
