## 一、用户体验层规则（UX）

> 本层规则面向**最终用户**的使用体验，旨在保证产品易用、清晰、可靠。所有 UI/UX 设计必须遵守。

### R1. ID 不可见
用户界面只展示可读名称，底层 ID 对用户完全不可见。
- ❌ 错误提示 "标签 ID #42 不存在"
- ✅ 错误提示 "标签「工作」不存在"

### R2. 点击优先 — 能点就不打，能滑就不点
除非必须输入文字或编辑内容，永远优先选择点击、滑动、选择器等前端交互。
- ❌ 让用户手动输入日期 "2024-06-14"
- ✅ 日期选择器，点两下搞定

### R3. 安全确认 — 不可逆操作必须二次确认
所有删除、清空、覆盖、退出等不可逆操作，必须弹出确认对话框并明确告知后果。
- ❌ 点击删除按钮，数据瞬间消失
- ✅ 「确认删除便签「买牛奶」？可在 12 小时内撤销。」→ [取消] [确认删除]

### R4. 即时反馈 — 每个操作必须有 UI 反馈
用户做了任何事，系统必须在 100ms 内给出可见反馈，不能让用户猜测"刚才到底成功了没"。
- ❌ 点击保存后页面毫无变化
- ✅ 保存按钮变灰 → 显示"已保存" SnackBar → 列表自动刷新

### R5. 空态处理 — 优雅处理空白和异常
所有列表、搜索结果、筛选条件都必须定义"空的时候显示什么"，不能留白或崩溃。
- ❌ 没有便签时显示一片空白
- ✅ 空状态插画 + "还没有便签，点击右下角 + 创建一个吧"

### R6. 幂等拒绝 — 重复操作给出明确拒绝
同一对象不能重复执行互斥操作，系统要明确返回失败并告知原因。
- ❌ 对已置顶便签再次置顶，系统无任何反应
- ✅ 弹出提示 "该便签已置顶"

### R7. 默认优先 — 替用户做最合理的选择
每个配置项必须有默认值，用户不设置也能正常使用。
- ❌ 首次打开要求先选择排序偏好才能进入
- ✅ 默认按更新时间排序，设置项自动持久化

### R8. 边界显式 — 所有数字约束必须显式定义
长度、数量、时间、容量等限制必须写在规格里，并配套超限处理策略。
- ❌ "标签名不要太长"
- ✅ "标签名 1–10 字符，超出时输入框禁止继续输入并提示「最多 10 个字符」"

### R9. 持久化 — 用户的选择必须记住
用户的偏好设置、排序方式、主题模式等必须本地持久化，下次打开自动恢复。
- ❌ 每次打开应用都重置为浅色模式
- ✅ 关闭前是夜间模式，下次打开仍是夜间模式

### R10. 可撤销 — 误操作必须可恢复
删除、修改等操作尽量提供撤销机制，而不是一删了之。
- ❌ 删除后数据永久消失
- ✅ 底部弹出 UndoBar「已删除 [撤销]」，保留 12 小时

### R11. 视觉过渡 — 状态变更必须定义动画
任何改变数据状态的操作，规格中必须同时描述动画/过渡效果和最终数据态。
禁止只写 `设置 is_completed=1` 而忽略 UI 如何平滑变化。
- ✅ "点击完成后，卡片执行 0.3s 透明度渐变 + 位移动画，动画结束后才刷新列表排序"

**补充 — 动画失败回退**：
若动画执行失败（如 Kivy 底层渲染异常），数据状态必须已变更，UI 可直接刷新到最终态，不依赖动画完成。动画是体验增强，不是功能依赖。

### R21. 不静默忽略 — 所有异常路径必须有 UI 反馈
用户任何输入即使不符合预期，系统也必须给出可见响应，禁止静默跳过。
- ❌ 用户输入不存在的标签名，系统无提示直接跳过
- ✅ Toast "标签「XXX」不存在，已自动跳过"

### R23. 错误信息用户可读
所有抛给用户的错误/警告/提示信息必须使用自然语言，说明「发生了什么 + 用户能做什么」，不得暴露技术细节（异常堆栈、数据库错误码、字段名、SQL 语句）。
- ❌ 错误提示 `sqlite3.IntegrityError: UNIQUE constraint failed: tag.name`
- ✅ 错误提示 "标签「工作」已存在，请使用其他名称"

### R31. 对齐优先同类型 — 对齐问题优先统一组件类型
同行两个组件垂直不对齐时，优先将两侧改为同类型组件（如都是 Label、都是 Button），而非调整 padding / pos_hint / y 偏移。
- ❌ `MDLabel` + `MDIconButton` 高度不一致 → 反复调 padding / 加 pos_hint 微调
- ❌ `MDLabel` + `MDFlatButton` 高度不一致 → 内边距不同，调了按钮又偏
- ✅ 两侧都用 `MDLabel`（文字渲染基线天然一致）

**Why:** 不同类型组件的内部布局策略不同（Label 无内边距 vs Button 有默认 padding，Icon 用图标字体基线 vs Label 用文本基线），微调偏移只能暴力对齐当前尺寸，换设备/换字体后又会错位。同类型组件共享同一套渲染规则，天然对齐。

### R32. 文字背景对比度 — 自定义组件必须检查主题兼容
任何自定义 UI 组件、浮层、提示框，只要涉及颜色属性（背景色、文字色），必须确保文字与背景在所有主题（浅色/深色）下均有足够对比度，肉眼清晰可见。

- ❌ 自定义 toast 背景用 `theme_cls.bg_darkest`（浅色模式≈浅灰白）+ 文字 `(1,1,1,1)`（纯白）→ 浅色模式下文字完全看不到
- ❌ 自定义弹窗用主题色背景但不验证深色模式下的文字颜色
- ✅ 优先使用内置组件（如 `MDSnackbar`），KivyMD 已处理好主题兼容
- ✅ 若必须自定义，用硬编码深色 `(0.12, 0.12, 0.12, 0.93)` + 白色文字，分别在浅色/深色主题下人工验证

**Why:** KivyMD 语义色（`bg_darkest`、`bg_dark` 等）在浅色/深色模式下颜色跨度极大，直接用作自定义组件背景时容易与文字融合。本次 toast 浅色模式下 `bg_darkest` ≈ 浅灰白 + 白色文字 → 完全不可见，改为内置 `MDSnackbar` 后零配置解决。

**How to apply:** 开发任何带颜色的自定义 UI 时，在浅色和深色两种主题下各验证一次。优先使用 KivyMD 内置组件（已适配主题），仅在功能不满足时才自定义。

### R33. 需求模糊时主动结构化追问 — UI 功能实现必须先澄清技术细节

当用户提出的需求涉及 UI 交互行为（尤其是没有报错信息、仅有主观体验描述的场合），且表达中存在模糊或歧义时，AI **必须**在动手修改前主动追问。追问必须满足三条标准：

1. **具体化** — 不能问"你期望什么行为"，而要给出 2–4 个互斥选项，每个选项附带具体场景描述
2. **结构化** — 将模糊描述拆解为可独立确认的技术维度（如：内容高度、滚动位置、可见性、可点击性、动画时长）
3. **附带术语解释** — 每个技术维度附带一句话白话解释，降低用户理解门槛

**示例 — 折叠已完成区场景**：

| 维度 | 白话解释 | 追问 |
|---|---|---|
| 内容高度 | 整个页面能滚多长 | 折叠后，下面留一段空白占位？还是页面直接缩短？ |
| 滚动位置 | 你当前看到的是页面哪一块 | 折叠瞬间，视线应该停在原位？还是跳回顶部？ |
| 可见性 | 眼睛能不能看见 | 折叠后已完成卡片是透明的还是彻底消失？ |
| 可点击性 | 手指点了有没有反应 | 折叠时虽然看不见，点到了还会触发操作吗？ |
| 响应时间 | 操作后多久看到变化 | 点击折叠按钮后立即变化？还是需要等？ |

**Why:** 用户知道"我想要什么结果"但往往不清楚技术术语（如"滚动视图"、"内容高度"、"绝对偏移"）的准确含义。AI 若直接按自己的理解动手，可能反复踩坑——第一轮用"保存/恢复 scroll_y"修复折叠跳动，第二轮用户才说清楚"高度不变、空白占位"。两轮 TDD 合计 11 个测试、3 次代码变更，根因是需求澄清不充分。如果一开始就逐维度确认，只需一轮即可完成。

**How to apply:**
1. 收到 UI 行为需求后，先自问：用户的描述涉及哪些技术维度？
2. 对每个维度，判断用户的用词是否有歧义
3. 若任一维度模糊，用表格形式逐项追问（选项 + 白话解释）
4. 用户确认所有维度后再开始写测试和改代码
5. 若用户用"我不知道技术细节，你帮我判断"回应，则给出推荐方案并说明理由，让用户只确认结果描述
